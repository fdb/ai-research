Inside an OpenType font

A font file is a collection of binary tables, organised so a layout engine can find a glyph from a character, measure its width, and trace its outline. This page parses one in front of you — every step, every byte.

Step 1 · The container

A font is a flat binary file

OpenType inherits its container format from Apple's TrueType, which is older than the web. The file has no compression, no versioning at the top, and no metadata block — just a four-byte tag, a count, and a series of fixed-width records pointing into the file. Everything else is found by following offsets.

Here are the first 64 bytes of the file, exactly as they sit on disk. The very first four bytes — the sfnt version — tell the renderer what to expect:

The sfnt tag distinguishes flavours. 0x00010000 means TrueType outlines (this file). 'OTTO' means Compact Font Format (CFF / CFF2) outlines — also valid OpenType, but with a different glyph payload. We'll only follow the TrueType path.

Step 2 · The table directory

An index of tables

Immediately after those 12 bytes comes the table directory: N records of 16 bytes each. Every record names one table — by a four-character ASCII tag like cmap or glyf — and gives its absolute offset and length in the file.

This is the only index in the file. To find any table later, we'll come back here.

    Tables marked in red are the ones we need to render a glyph from a character. The rest — kerning, signatures, OS-specific metrics — are out of scope.

    Step 3 · maxp · the glyph count

    How many glyphs does the font have?

    The simplest table answers the most basic question. maxp ("maximum profile") begins with a version number followed by a single uint16: the number of glyphs.

    Every other table that lists per-glyph data — outlines, advance widths, locations — has exactly this many entries. (Or, in the case of hmtx, fewer, with a special rule we'll see in a minute.)

    The version 0x00010000 means this is the longer "TrueType" maxp — it contains additional limits like the deepest hint stack the renderer needs to allocate. For our purposes, only numGlyphs matters.

    Step 4 · head · the design grid

    The coordinate system the font lives in

    Glyphs are not stored in pixels. They are stored as integer coordinates on an abstract grid, the em square, sized by the unitsPerEm field. To draw a glyph at a given size, you scale by pixelsPerEm / unitsPerEm.

    The head table also tells us the format of the loca table we'll need later — short or long — and the bounding box that contains every glyph in the font.

    Step 5 · hhea + hmtx · horizontal metrics

    How wide is each glyph, and where does the line sit?

    For horizontal scripts, hhea sets the page-level metrics — ascent, descent, line gap — and tells us how big the per-glyph metrics array hmtx is.

    Then hmtx stores one (advanceWidth, leftSideBearing) pair per glyph — almost. As a space optimisation, only the first numberOfHMetrics glyphs get a pair; the remaining glyphs share the last advance width, with their own leftSideBearings packed into a tail array.

    Below: the first few entries. Pairs are 4 bytes each (uint16 advance, int16 lsb).

    Step 6 · cmap · characters become glyphs

    Mapping a Unicode character to a glyph index

    None of the tables we've seen so far know about characters. They all index by glyph index: a number from 0 to numGlyphs − 1. The job of cmap is to translate Unicode codepoints into those numbers.

    A font carries multiple subtables, one per platform/encoding pair. We'll pick the Unicode format 4 subtable — the one used for the Basic Multilingual Plane on every modern system.

    Subtables on offer

    Format 4: segment mapping to delta values

    Format 4 is a clever data structure for sparse maps. The codepoint space is partitioned into segments of contiguous codepoints. For each segment we store a startCode, endCode, an idDelta and an idRangeOffset. Lookup is a binary search for the segment containing the codepoint, followed by either a simple addition (when idRangeOffset is zero) or an indirection through a glyphIdArray.

    Step 7 · loca + glyf · outlines

    Drawing the glyph

    We finally arrive at the shapes. loca ("locations") is just an array of offsets, one per glyph plus a sentinel at the end — so the bytes for glyph n are glyf[loca[n] … loca[n+1]). The format of those offsets is determined by the indexToLocFormat field we read earlier.

    Each glyph record begins with a header:

    • int16 numberOfContours — positive for a simple glyph, −1 for a composite
    • int16 xMin, yMin, xMax, yMax — the glyph's bounding box

    For a simple glyph the rest is: end-points-of-contours, instructions, then run-length-encoded flags, then x and y deltas. The flags are where the magic is — bit 0 says whether each point is on the curve or off. Quadratic Bézier control points are off-curve. When two off-curve points sit next to each other the renderer is required to imagine an on-curve point exactly between them.

    Putting it together

    From a character to a shape

    To render the letter you typed:

    1. Read the cmap subtable, find the segment containing the codepoint, follow the formula to get a glyph index.
    2. Read the indexToLocFormat from head.
    3. Use that to read loca[i] and loca[i+1] — the start and end offsets into glyf.
    4. Parse the glyph header, then flags, then x and y deltas. Build the contours.
    5. Read the advanceWidth and leftSideBearing for that glyph from hmtx.
    6. Read ascender, descender, lineGap from hhea for line layout.
    7. Scale everything by pixelsPerEm / unitsPerEm from head.
    8. Draw the contours, treating off-curve pairs as quadratic Béziers with implicit on-curve midpoints.

    That's the whole rendering pipeline for a TrueType-flavored OpenType font, minus hinting (which fine-tunes pixel positions at small sizes), kern/GPOS (which adjusts spacing between glyph pairs), GSUB (which substitutes ligatures and contextual forms), and the variable font axes — each large enough for its own page.
