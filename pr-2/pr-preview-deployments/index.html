PR Preview Deployments

Two GitHub Actions workflows that publish a live preview for every pull request to a pr-<num>/ path on the gh-pages branch, then clean up when the PR closes.

The flow

push to main         pull_request
     │                    │
     ▼                    ▼
  deploy.yml      pr-preview.yml
     │                    │
     ▼                    ▼
        gh-pages branch
     │                    │
     ▼                    ▼
  /index.html       /pr-42/index.html

One publish branch, two workflows, disjoint subtrees: the main deploy excludes pr-*/ from its clean step so previews survive every push to main; the preview deploy writes only to pr-<num>/.

What's in this folder

README.md notes.md

The actual workflows live in .github/workflows/ at the repo root — see deploy.yml & pr-preview.yml on GitHub.

Highlights

  • Built-in GITHUB_TOKEN only — no secrets to configure.
  • Sticky PR comment, edited in place rather than re-posted on every push.
  • Per-PR concurrency group with cancel-in-progress, so rapid pushes don't pile up.
  • Fork PRs are skipped instead of failing — pull_request_target isn't worth its security cost for a static preview.

See the README for setup, behaviour, and trade-offs.
